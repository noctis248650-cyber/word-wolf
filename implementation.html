워드울프 시스템 구현 설명서

GitHub Pages로 배포되는 정적 웹앱, Supabase Postgres/RPC 기반 실시간 게임 상태 관리, Supabase Edge Function과 OpenAI Responses API를 이용한 LLM AI 플레이어 구조를 정리한 문서입니다.

V1 실행 V2 실행 플레이 설명서

Overview

전체 구조

클라이언트는 게임 화면과 사용자 입력을 담당하고, 서버 권한 로직은 Supabase 쪽에 둡니다.

Browser UI GitHub Pages 정적 파일. 방 목록, 채팅, 힌트, 투표 UI 렌더링.
Supabase JS Client anon public key로 RPC와 Edge Function을 호출.
Postgres RPC 방 생성, 입장, 단계 전환, 투표, 결과 계산을 DB 함수에서 처리.
Edge Function AI 플레이어 행동을 중계. OpenAI 키는 여기에서만 사용.
OpenAI API 힌트, 채팅 답변, 투표 판단, 최종 추리 텍스트 생성.

Game State

게임 단계 플로우

단계 전환은 클라이언트 타이머가 아니라 Supabase RPC에서 현재 시간을 기준으로 결정합니다.

Lobby 방 생성, 입장, 준비 완료, AI 추가
Reveal 10초 동안 각자 단어 확인
Hint 플레이어당 30초 힌트 제출
Chat 180초 자유 채팅
Vote 30초 안에 울프 지목
Wolf Guess / Result 울프 최종 추리 후 결과 공개
시민이 투표로 울프를 잡아도 바로 끝나지 않고, 울프가 시민 단어를 맞히면 역전 승리합니다. 결과에는 승리 팀, 워드울프, 시민 단어, 울프 단어, 울프 입력값, 투표 요약이 공개됩니다.

Data Model

주요 데이터 구조

방 하나의 진행 상태는 ww_rooms.current_game JSON에 모아 저장합니다.

테이블/파일 역할 주요 내용
ww_rooms 방과 현재 라운드 상태 방 코드, 방장, 플레이어 목록, 설정, 현재 단계, 라운드, 단어 배정, 힌트, 투표, 결과를 저장합니다.
ww_messages 방 채팅 방 코드별 최근 채팅을 저장합니다. RPC는 권한 확인 후 최대 80개를 시간순으로 반환합니다.
ww_word_pairs 워드울프 단어 풀 시민 단어, 울프 단어, 큰 카테고리를 CSV에서 동기화합니다.
ww_word_pair_history 최근 단어 반복 방지 최근 20개 단어쌍을 제외하고 랜덤 추출하며, 히스토리는 500개까지만 유지합니다.
supabase/word_pairs.csv 단어 원본 코드와 분리된 데이터 소스입니다. GitHub Actions가 Supabase 테이블로 동기화합니다.

LLM Players

OpenAI 기반 AI 플레이어 구현

AI는 실제 플레이어처럼 힌트, 채팅, 투표, 최종 추리를 RPC로 제출합니다.

모델과 호출 방식

  • Supabase Edge Function ai-bot-turn이 OpenAI Responses API를 호출합니다.
  • 모델은 OPENAI_MODEL 환경변수로 교체 가능하며, 코드 기본값은 gpt-4.1입니다.
  • 각 요청은 temperature: 0.8로 생성 다양성을 줍니다.
  • 출력 토큰은 행동별로 40~90 정도로 제한해 짧은 답변을 유도합니다.
  • OPENAI_API_KEY는 Edge Function secret에만 저장되고 브라우저에는 노출되지 않습니다.

AI 플레이어 이름

AI 추가 시 고정된 이름 풀을 순서대로 사용합니다.

AI Sequence

AI 행동 처리 순서

AI도 DB 상태를 직접 조작하지 않고, 사람 플레이어와 같은 RPC 경로를 사용합니다.

클라이언트가 AI 차례를 감지

방장 브라우저가 현재 단계와 활성 플레이어를 보고 AI 힌트, 투표, 최종 추리 차례인지 판단합니다.

Edge Function 호출

db.functions.invoke("ai-bot-turn")으로 방 코드, 방장 ID, AI 플레이어 ID, 행동 종류를 전달합니다.

권한과 관점 검증

Edge Function은 방장이 호출했는지, 대상 플레이어가 실제 AI인지 확인하고 AI 자신의 관점으로 방 상태를 조회합니다.

프롬프트 구성

AI의 단어, 역할, 카테고리, 기존 힌트, 최근 채팅을 합쳐 행동별 프롬프트를 만듭니다.

OpenAI 응답 생성

Responses API가 힌트 문장, 채팅 답변, 투표 대상, 시민 단어 추리를 생성합니다.

RPC로 결과 반영

생성된 결과는 ww_submit_hint, ww_send_message, ww_vote, ww_submit_wolf_guess를 통해 저장됩니다.

Prompt Design

AI 답변 품질을 위한 제어

워드울프 게임 특성상 “너무 직접적인 힌트”와 “단어 유출”을 막는 로직이 중요합니다.

힌트 생성

  • 정답의 사전적 특징 대신 한 단계 떨어진 연상을 요구합니다.
  • 문화, 장소, 경험, 브랜드, 말장난, 분위기, 용도 주변부 같은 넓은 힌트를 허용합니다.
  • 시민 단어와 울프 단어를 금지어로 두고, 생성 답변에 금지어가 들어가면 최대 3회 재시도합니다.
  • 재시도 후에도 실패하면 안전한 fallback 힌트를 사용합니다.

채팅 반응

  • 최근 채팅 18개와 방금 메시지를 함께 전달합니다.
  • 질문에는 질문에 답하고, 의견에는 그 의견에 이어 말하도록 지시합니다.
  • 정답 단어 직접 공개와 노골적인 힌트는 피하도록 지시합니다.
  • 출력은 친구 채팅방처럼 짧은 1문장으로 제한합니다.

투표 판단

  • AI 자신의 단어와 역할을 기준으로 힌트, 최근 채팅을 분석합니다.
  • 후보 목록은 플레이어 ID와 이름으로 제공합니다.
  • 모델 출력이 ID 또는 이름과 매칭되면 해당 플레이어에게 투표합니다.
  • 매칭 실패 시 게임이 멈추지 않도록 후보 중 랜덤 fallback을 사용합니다.

울프 최종 추리

  • 지목된 AI 울프만 실행됩니다.
  • AI가 받은 울프 단어, 카테고리, 힌트, 최근 채팅을 기반으로 시민 단어를 추리합니다.
  • 결과는 단어 하나만 제출하도록 제한합니다.

Mention Routing

이름 호출 기반 AI 채팅 라우팅

여러 AI가 동시에 답하지 않도록 브라우저 쪽에서 호출 대상을 정합니다.

사람 채팅 발생 가장 최근 사람 메시지를 감지합니다.
이름 prefix 검사 공백 제거 후 메시지가 AI 이름으로 시작하는지 확인합니다.
이름이 있으면 예: “조지 이거 뭐야?” → 조지만 답변합니다.
이름이 없으면 AI 중 한 명만 순번으로 답변합니다.
중복 방지 메시지 ID와 AI ID 조합으로 이미 답한 메시지는 다시 처리하지 않습니다.

Security

키와 권한 경계

포트폴리오에서 강조하기 좋은 부분은 “비밀 키가 브라우저로 가지 않는다”는 점입니다.

브라우저에 있는 것

  • Supabase Project URL
  • Supabase anon public key
  • 플레이어 ID, 방 코드, 아이콘 같은 로컬 세션 정보

브라우저에 없는 것

  • OpenAI API Key
  • Supabase service role key
  • Database connection string
  • Supabase access token
직접 테이블 조회 대신 security definer RPC를 사용합니다. 특히 방 상태 조회 함수는 viewer ID를 기준으로 해당 플레이어의 단어만 내려주기 때문에, 시민과 울프의 비밀 단어가 브라우저에 한꺼번에 노출되지 않습니다.

Portfolio Summary

구현 포인트 요약

기술적으로 보여줄 수 있는 점

  • 정적 웹앱에서도 Supabase RPC를 활용해 실시간 멀티플레이 게임을 구성했습니다.
  • 게임 규칙과 승패 판정을 DB 함수에 두어 클라이언트 조작 가능성을 줄였습니다.
  • OpenAI API는 서버리스 함수에서만 호출하여 API key 노출을 막았습니다.
  • LLM 출력을 그대로 믿지 않고 금지어 검사, fallback, 중복 응답 방지 로직을 추가했습니다.

확장 가능성

  • 모델은 OPENAI_MODEL 환경변수만 바꿔 쉽게 교체할 수 있습니다.
  • 단어 DB는 CSV로 관리되어 대량 추가와 검수 자동화가 가능합니다.
  • AI 성격별 프롬프트, 난이도별 힌트 스타일, 채팅 moderation 같은 기능을 추가하기 쉽습니다.
  • V1/V2처럼 같은 게임 로직 위에 다른 UI를 얹는 구조로 실험할 수 있습니다.
